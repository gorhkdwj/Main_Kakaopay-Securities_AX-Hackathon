### D-0716-0210-main · 온보딩 고지 + 브리핑 옵트아웃("간단 모드") + 보편 최소선
*(구 `docs/plans/2026-07-17_제품결정로그.md`의 PD-01을 내용 그대로 이전 — 원 기록 시각 기준 ID 부여)*

**상황**
판단 여권의 브리핑 흐름(계획 회상→사실→체크리스트→시나리오)이 "주문 체결까지 너무 오래 걸린다, 빠른 대응을 못 한다"는 인상을 줄 수 있음. 방어 논리(체결이 아니라 확신 없는 결정을 늦추는 것, '빠른 대응'의 실증적 해악 — 첫날 매도율 41%/22%, 추격매수 -3.1%, 과다거래 11.4% vs 17.9%, 급한 손절은 스탑로스로 사전 자동화, 장애 순간은 안전모드가 보호)는 성립하나, 마찰로 인한 이탈은 실측 전 리스크(가설 H4)로 남음.

**검토한 선택지**
① 브리핑 흐름 강제 ② 완전 옵트아웃(추적 없음) ③ 온보딩 고지 + 모드 선택 + 최소선 유지 + 조건부 재제안

**결정** — ③ 채택. 세부:
1. **온보딩 고지**: 첫 사용 시 브리핑 흐름의 시간 비용(2~3분, 재사용 시 축약)과 그 이유(비가역 결정 전 확인)를 숨기지 않고 고지한다.
2. **옵트아웃 허용 — 모드 선택 프레이밍**: "초보자 기능 끄기"가 아니라 **"간단 모드 / 브리핑 모드" 선택**으로 표현한다(낙인 금지 — 단일 초보/숙련 라벨 금지 원칙, X-CLM-0002).
3. **간단 모드의 화면**: 기존 주문 흐름에 **매수·매도 전 비용·세금·D+2 출금일·비가역성 미리보기(F4)를 보완**한 화면을 제공한다. F4는 단독 80점·규제 🟢·국내 미확인 공백으로 전 사용자 보편 가치가 확인된 레이어다(C-CLM-0411=X-CLM-0017, 리서치 §2.4·§3.1).
4. **옵트아웃 불가 최소선**: 비용·세금·D+2·비가역성 고지와 주문 확인 화면은 어떤 모드에서도 유지한다(금소법 §19 설명의무 취지 C-CLM-0509, VOC 사후 충격 C-CLM-0405·0303, 전금법 사전 예방 우선 C-CLM-0512).
5. **조건부 재제안**: 옵트아웃은 영구 차단이 아니다. 행동취약 신호(생애 첫 매도, 급락일 보유 종목 -N% 도달, 사전 감수손실 한도 접근) 발생 시 **1탭 재제안 배너**를 노출하되 강요·반복 알림은 금지한다. 근거: 과잉확신형(지식 낮음+자신감 높음)은 자기 선언 분류에서 가장 먼저 이탈하는 집단이며 도움 필요도는 가장 높음(X-CLM-0002, C-SRC-0214).
6. **양방향 동일 접근성**: 모드 전환은 설정에서 상시·동일 난이도로 가능(다크패턴 금지).

**근거** — 위 각 항 병기 (리서치 대장 ID 연결)

**영향** — 온보딩·모드 전환 구현은 후순위(본선 데모 범위 외). 심사 Q&A 답변 지참: "숙련 사용자는 온보딩에서 간단 모드를 선택할 수 있습니다. 다만 비용·세금·출금일 미리보기는 숙련자에게도 국내 앱에 없는 가치라 전 모드 공통이고, 급락일·첫 매도 같은 취약 순간에만 브리핑을 가볍게 다시 제안합니다 — 강요는 없습니다."

**재검토 조건** — 파일럿 지표: 옵트아웃률(온보딩 vs 30일 후), 재제안 수락률, 간단 모드 사용자의 정산·출금 문의율, 브리핑 의사결정 시간·중도 이탈률(H4) — 이탈 과다 시 "접힌 기본+펼침 선택"으로 완화.

### D-0716-0911-main · Claude·Codex 이중 헌법과 동기화 규칙

**상황**

- ③을 채택한다. `CLAUDE.md`와 `AGENTS.md`는 동일한 공통 작업 헌법을 유지한다.

질문 착지 정리(D-0718-0028) 후 사용자 제안: "AI가 연결되어도 LLM이 생성할 수 없는 정보를 데모로 보여줄 사전 입력 답변 보기 방식이 있으면 좋겠다". 문제의 뿌리는 LLM 능력이 아니라 데이터 부재(실적 발표 일정 등이 fixture에 없어 unknowns로만 존재).

**검토한 선택지**
① 새 보기 방식(사전 입력 답변 모드 — UI 모드 추가) ② **가상 fixture 확장(IR 일정 공시 추가 — 기존 구조 흡수)** ③ 구현 없이 발표 답변만. 논의 후 사용자 지시: ② 진행.

**결정** — ② 채택. 시나리오당 disclosures에 IR 일정 공시 1건 추가(끝자리 x03 — 계약 §2 규약 신설): loss8 "4분기 잠정 발표 예정일 2026-10-29"(DEMO-SRC-103) · profit15 "2분기 확정 2026-08-13"(DEMO-SRC-203) · first_buy "2분기 확정 2026-08-12"(DEMO-SRC-303). **표현 규약**: "회사가 예정일을 공시했다"는 과거 사실 서술만 — 미래 발표를 확정 사실처럼 쓰지 않는다(§14 "미래 시각을 예정 사실처럼 쓰지 않는다"와 정합). 수급 주체·업계 비교는 의도적으로 남는 모름(정직 서사 유지). 스키마 변경 없음(disclosures 다건은 기존 계약 §3.1 형태). 적용 범위: 데모 3종 fixture 전용 — 실서비스에서는 공시·일정 연동 데이터가 이 자리를 대체.

**근거** — ① 새 UI 모드는 정직성 원칙(미구현 기능 표시 금지)과 긴장 + 상태 복잡도·검증 사이클 부담 ② 데이터를 제공하면 LLM이 실제로 인용 가능("생성 못 하는 정보"의 근본 해소)하고 출처·기준시각·guard·캐시 지문 등 기존 안전 구조에 그대로 흡수 ③ 서사 강화: "데이터가 연결되는 만큼 모름이 출처 있는 사실로 바뀌고, 그래도 남는 모름은 정직하게 모름"(unknowns가 "10-29 발표 이후 확인 가능"으로 일정 사실과 결합).

**영향** — 계약 §2(x03 규약)·§3.1(disclosures 2건), fixture 3종, 캐시 3종 재생성(지문 갱신 — 사실 4카드·unknowns 재구성), 프롬프트(외부 정보 예시에서 '실적 발표 일정' 제거 + 내부 용어 '<data>' 화면 노출 금지 규칙 — profit15 재생성으로 노출 1건 해소), 골든 테스트 6곳 갱신(source 순서 3열·카운터 3→6·차단 시 잔존 2카드), 289 tests·게이트·브라우저 통과.

**재검토 조건** — 실서비스 공시·일정 연동 시 이 가상 공시 슬롯을 실데이터로 대체, fixture 재수정 시 캐시 재생성 필수(지문), 심사 Q&A에서 "일정 공시도 가상인가" 질문 시 "전부 교육용 가상 데이터이며 화면에 (가상) 병기" 답변.


### D-0718-0210-main · S0 인터셉트 발동 방식 개정 — 클릭 발동 + 중앙 모달

**검토한 선택지**

- goStep은 팝업을 항상 닫고, 발동 경로는 S0 구매/판매 버튼 클릭이 유일(로드·전환·"처음으로" 복귀 시 미자동).

**근거** — ① 클릭→인터셉트의 인과를 시연자가 직접 보여줘 재현 서사가 강화됨(자동 열림은 인과가 생략된 채 시작) ② S0 주문 화면 재현이 팝업에 가려지지 않고 온전히 먼저 보임 ③ 진입 갈림길은 '주문 확인'(하단 시트 관례)이 아니라 '선택 대화상자' — 중앙 모달이 역할에 맞음.

**영향** — 계약 §9(상시 인터셉트 항), 스펙 §0(통합 시점)·§4(S0 행), project-plan §3.1, validation-plan I-01, app.js(loadScenario·goStep), app.css(백드롭 분리·모달 룩). DOM 테스트는 클릭 배선만 단언해 무수정 통과(289). 시연 대본 유의: 시연 시작 시 발표자가 구매/판매 버튼을 눌러 팝업을 발동한다(S7 리허설 체크 항목).
