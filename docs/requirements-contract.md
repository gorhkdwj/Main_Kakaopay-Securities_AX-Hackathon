# 기준 계약 문서 · 판단 여권 (Decision Passport)

> **지위**: 프로젝트 전반의 모든 지표·수식·스키마·표기·판정의 **단일 기준(single source of truth)**. 코드·화면·테스트·문서·브리프의 어떤 수치도 이 문서와 달라선 안 된다.
> **변경 절차**: 기준이 바뀌면 ① 이 문서를 먼저 갱신하고 ② Decisionlog에 이유를 남긴 뒤 ③ 코드·테스트를 맞춘다(헌법 §5). 코드를 먼저 고치는 것을 금지한다.
> **데모 고정값**: `[데모 고정]` 표시는 본선 데모를 위해 임시 확정한 값이다(근거: Decisionlog `D-0716-0923-main`). 실서비스 전 원문·정산 규칙 재확인 대상이며, §13에 목록화한다.

## 1. 용어·엔티티 정의

| 용어 | 정의 |
|---|---|
| 종목(instrument) | 데모에서는 **가상 종목만** 사용(예: 가온전자). 실존 종목 사용 금지(권유 오해 방지 — 헌법 §7) |
| 보유(position) | {종목, 보유수량, 평균 매수가} — 평균 매수가는 fixture 고정값(매수 이력 재계산 없음, 데모 단순화) |
| 계획(plan) | 사용자가 기록한 {목표 기간, 감수 가능 손실률, 재검토 조건, 기록일} |
| 스냅샷(snapshot) | 실제 수집에 성공해 **동결**된 시장 데이터. 수집 성공 확인 전에는 존재하지 않는 것으로 취급 |
| fixture | 오프라인 시연용으로 고정한 가상 시나리오 데이터(최종 폴백) — 스냅샷과 **혼용 금지**: 가상 종목의 가격·공시는 항상 fixture가 원천이고, 스냅샷은 시장 상황 카드·수집 증빙에만 사용 |
| 세션(session) | 사용자가 브리핑 시작~회고 기록까지 진행한 1회 흐름 — 안전 지표의 집계 단위 |
| 모의 체결 | 실주문 없이 로컬 엔진이 생성하는 가상 체결 기록. 화면 전역에 "모의" 상시 표기 |
| as_of(기준시각) | 해당 데이터가 유효한 시각. **미래 시각·미수집 시각을 쓰지 않는다**(F-CLM-0099) |

## 2. ID 체계

| ID | 형식 | 발급 주체 | 용도 |
|---|---|---|---|
| source_id | 리서치 근거 = 병합 대장 ID(`F-/C-/X-SRC-####`) / 데모 가상 출처 = `DEMO-SRC-###` / **실데이터 출처 = `YF-SRC-###`(D-0718-1110)** | fixture 작성 시 고정 | facts의 출처 추적. **DEMO-SRC는 가상 라벨임을 화면에 "(가상)"으로 병기**. 부여 규약: 백의 자리 1=가온전자·2=한빛식품·3=다온소재, 끝자리 01=실적 공시(fixture disclosures)·02=시세·거래량(값의 원천은 fixture price/volume — 화면 조립부가 사용)·03=IR 일정 공시(다음 실적 발표 예정일 — D-0718-0107-main: '모름'을 출처 있는 사실로 승격하는 데이터 확장). **YF-SRC 규약(real_005930)**: 001=52주 고저·002=거래량 vs 평균·003=섹터/산업·004=공시된 실적 예정일·005=시세 카드(price/volume 원천)·006=시장 맥락 지표(market_context 스냅샷 — 동반자 대화 인용, `[데모 고정]`) — 원천은 전부 Yahoo Finance 동결 스냅샷, 화면에 지연 고지 병기 |
| calculation_id | `CALC-YYYYMMDD-HHMMSS-<seq>` | engine | 계산 결과 ↔ AI 응답 ↔ 화면 표의 연결. LLM 출력의 숫자는 반드시 이 ID의 결과에서만 인용 |
| decision_record_id | `REC-<세션>-<seq>` | webapp | 사후 회고(판단 기록) 식별 |
| 운영 로그 ID | `W-/D-/T-MMDD-HHMM-세션` | 각 세션 | 헌법 §11·§13 |

### 3.1-b 실종목 시나리오 fixture — `data/fixtures/scenario_real_{ticker}.json` (D-0718-1110-main)
§3.1과 동일 스키마를 따르되 아래 델타만 다르다. **원천**: yfinance 스냅샷 1회 수집·동결(§3.2) → 어댑터 `scripts/data/build_real_scenario.py`가 변환(수기 숫자 입력 금지 — 계산·생성 분리).

| 델타 항목 | 규칙 |
|---|---|
| scenario_id | `real_{티커숫자}` (예: `real_005930`) |
| is_synthetic | **false** — 이때 `data_origin` 필수: {source("Yahoo Finance via yfinance"), collected_at(KST), delay_note("거래소 시세 지연(최대 20분) — 실시간 아님"), snapshot_ref(원천 스냅샷 파일명)} |
| 화면 배지 | 가상("교육용 가상 데이터")과 구분: **"교육용 모의 환경 — 실데이터 스냅샷 기준"**. 보유·계획·예수금 등 계좌성 값은 교육용 모의 값임을 병기 |
| instrument.code | 실제 티커(`005930.KS` 등). market은 접미사 매핑(.KS→KOSPI, .KQ→KOSDAQ) |
| price / price.history / volume | 스냅샷 실측값. closes는 최근 실종가 시계열(목표 250, 부족 시 실길이 ≥ 30 허용). 정합 규칙(closes[-1]==close·closes[-2]==prev_close·전 원소 양수·미래 시각 금지)은 §3.1과 동일 |
| disclosures[] | 실공시 원문은 yfinance 미제공 — 시세·52주 고저·거래량 대비·섹터 등 **측정 사실 카드**로 대체 가능(각 source_id·as_of 필수). 실적 예정일은 공시된 예정일만 "예정일이 공시되어 있다" 서술로(§14 정합) |
| source_id | 실데이터 출처 ID `YF-SRC-###`(§2 확장) — 정책 가드 SRC-FMT 패턴과 정합해야 하며, 패턴 확장 시 가드 테스트 동반 |
| **금지** | 권유성 필드(recommendation\*·target\*·upgrades_downgrades·애널리스트 의견 등)는 **수집·저장·표시 전부 금지** — 어댑터·수집기의 화이트리스트가 강제(리서치: `docs/research/2026-07-18_yfinance_활용판단.md` §2(d)) |
| holding / plan / cash / past_records | 교육용 모의 값 `[데모 고정]`(시연 서사용) — 평단 등은 스냅샷 시계열 내 실제 과거 종가에서 선택 |
| trade_date | 스냅샷 기준 최근 거래일 |
| 백업 | 가상 3종(§3.1)은 오프라인 백업 경로로 보존 — 네트워크·수집 실패 시 기본 폴백(§8 장애 안전) |

### 3.3 판단 기록 — `out/records/REC-*.json` (Git 제외)
{record_id, scenario_id, intent(방향별 4택 — §9), reason_text, calculation_id?, created_at, review_date?} — 사후 회고 화면의 원천. 결과(수익률)를 저장하지 않는다(과정 중심 원칙). intent 검증: §9의 고정 라벨 집합 + **구매 수량 의향은 `N주 구매 검토`(N=1~999) 동적 라벨 허용**(④ 구매 수량 조정 — D-0718-0255).

## 4. 주요 지표 정의 (정의가 곧 코드 명세)

| 지표 | 정의 | 주의 |
|---|---|---|
| 평가손익 | (현재가 − 평균 매수가) × 보유수량 | 매도 전 상태 표시용. 실현손익과 혼용 금지 — 화면 라벨 구분 |
| 평가손익률 | (현재가 − 평균 매수가) ÷ 평균 매수가 × 100 | 소수 1자리 반올림 표기 |
| 매도대금 | 기준가 × 매도수량 | 기준가 = fixture close(지정가 가정 — §5) |
| 실현손익 | 예상수령액 − (평균 매수가 × 매도수량) | **비용 차감 후** 정의(비용 미포함 손익과 구분) |
| 잔여 비중 | 매도 후 해당 종목 평가액 ÷ **portfolio_total_value(매도 전 스냅샷 고정)** × 100 | [데모 고정] 분모에 매도대금 재편입 없음 — 단순화 한계를 §13에 명시. 소수 1자리 |
| 매수 후 비중(K2) | (기존 보유수량 + 매수수량) × 현재가 ÷ **portfolio_total_value(총자산 고정)** × 100 | [데모 고정] **분모는 매도 잔여 비중과 동일한 총자산 고정 — 양방향 대칭(D-0718-0335)**. 분자는 **매수 후 이 종목의 총 평가액**(기존 보유 평가 + 이번 매수 평가)이다 — 보유가 있는 시나리오의 매수 검토에서 기존 보유를 누락하던 오류 수정(구 "이번 매수분 ÷ cash"는 first_buy 전용 정의가 보유 시나리오에 잘못 적용된 것). first_buy는 보유 0·전액 현금이라 46.0% 불변. >40%면 집중도 경고(§5.3). 소수 1자리 |
| 결제일(D+2) | trade_date + 2영업일(토·일 제외) | [데모 고정] 공휴일 미반영 — 데모 기간(7/17~21) 공휴일 없음 확인됨 |
| 안전 지표 | §10 참조 | 분모 정의 필수 |

## 5. 계산 계약 (수식 · 데모 고정값 · 골든값)

### 5.1 수식
- **수수료** = ⌊매도대금 × 0.00015⌋ — [데모 고정] **KRX 온라인 0.015% 단일 적용**. 근거: 데모 시나리오는 국내주식이고, SOR 배분 비율은 공개 원문으로 확인 불가하여 지어낼 수 없으며, KRX 요율(0.015%)이 NXT(0.014%)보다 높아 보수적. NXT 요율은 규칙 엔진 확장 슬롯으로 명시(원천: 회사 수수료 안내 C-SRC-0130, 2026-07-16 열람).
- **거래세·농특세** = ⌊매도대금 × 0.0020⌋ — **2026년 현행 국내 상장주식 매도 합계 0.20% 확인됨**: 코스피 = 증권거래세 0.05% + 농어촌특별세 0.15%, 코스닥 = 증권거래세 0.20%(농특세 없음) — 어느 쪽이든 합계 0.20%로 동일. 데모 종목은 전부 코스피 상장 가정, 화면 라벨은 "거래세·농특세". **연혁 주의**: 2025년 0.15% → 2026-01-01부터 0.20%로 상향(연도별 변동 — §13 대장에서 효력일 관리). 근거: 한국세정신문 「내년 1월부터 증권거래세율 코스피 0.05%, 코스닥 0.20%로 상향」(https://taxtimes.co.kr/news/article.html?no=272624, 2025-12 보도) 외 복수 언론·증권사 공지, 확인일 2026-07-16. 시장·상품 일반화 금지(CV-15) 유지: 코넥스 0.10%·비상장 0.35% 등 상이. 매수에는 거래세·농특세 없음(K2 시나리오).
- **예상수령액** = 매도대금 − 수수료 − 거래세·농특세.
- **체결 후 평균 구매가(예상, D-0718-0255 후속 — 사용자 요청 2026-07-18)** = 매수: ⌊(기존 보유수량×평균 구매가 + 매수대금) ÷ (기존 보유수량+매수수량)⌋ — **순수 매입가 기준(수수료 미포함 — 실현손익의 avg_price와 동일 기준)·원 미만 절사 [데모 고정]**. 보유 0이면 = 기준가. 매도: **평균 구매가는 변하지 않는다**(잔여 보유 시 기존 값 유지 표시·전량 매도 시 잔여 없음 — null). ④ 비교 표에 표시. 골든값: loss8 10주 매수 49,000 · 8주 매수 49,157(절사) · profit15 10주 매수 42,000 · first_buy 46,000(=기준가).
- **반올림** = 수수료·거래세·농특세 **각각 원 미만 절사(내림)** 후 차감 — [데모 고정] 실제 정산 규칙 원문 미확인, §13 재검토 항목.
- **주문 유형** = 데모는 지정가(기준가=close) 가정. 시장가 표시 시 "실제 체결가는 달라질 수 있음" 고지 필수(이해 확인 문항과 연동).
- **체결시장 표기** = "KRX/NXT 중 유리한 시장으로 자동 배분(SOR)" **정보 문구로만** 표시하고 계산은 KRX 기준 — 배분 비율을 지어내지 않는다.

### 5.3 경고·오류 임계값 [데모 고정 — D-0716-0923-main]
- 매도: 수량 ∉ [1, 보유수량] → **오류**(계산·체결 생성 금지). 전량 매도 선택 시 "보유 전량입니다" 라벨 표시.
- 매수(K2): **총 결제예정액(매수대금+수수료)** > 가용 예수금(cash) → **오류**(계산·체결 생성 금지). 매수 후 비중(§4 — 분모: 매수 전 예수금 총액 고정) > **40%** → **집중도 경고**(차단 아님 — 정보 표시). 골든 경계: §5.2-c(21주 가능·22주 오류).
- 수량은 양의 정수만(소수점 거래는 데모 범위 외 — 미지원 명시).

## 6. AI 응답 계약 (모든 LLM 출력의 유일한 형식)

```json
{
  "facts":            [{"text": str, "source_id": str, "as_of": str}],
  "interpretations":  [{"text": str, "basis": [str], "stance": "긍정 시각|부정 시각"}],
  "unknowns":         [str],
  "user_inputs":      {"quantity": int|null, "intent": str|null, "situation": str},
  "calculation_id":   str|null,
  "policy_result":    "information_only|blocked_partial|error",
  "next_questions":   [str]
}
```

| 필드 | 검증 규칙 (guard가 렌더링 전 강제) |
|---|---|
| facts[].source_id / as_of | **필수** — 없으면 해당 fact 블록 차단. source_id는 §2 체계에 실재해야 함 |
| facts[].text | 숫자 포함 시 calculation_id 결과 또는 fixture 원천값과 일치해야 함(LLM 산수 금지) |
| interpretations | 사실과 분리 렌더("해석 — 사실이 아니에요" 라벨). stance 양면이 모두 있어야 정상(한쪽만이면 "반대 시각 확인 안 됨" 명시 요구) |
| unknowns | 최소 1개 권장("내일 가격은 알 수 없음" 등) — 비어 있으면 경고 로그 |
| user_inputs.quantity | 화면 입력값과 일치 — 불일치 시 응답 전체 차단(수량 조작 방지) |
| calculation_id | 숫자 인용이 있으면 필수 |
| next_questions | **착지는 서비스 안**(사용자 결정 2026-07-18, D-0718-0028-main): 제공된 데이터(내 투자 계획·지난 일지·공시 원문·시세)나 이후 단계 화면(비용·세금·D+2 안내, 수량별 시나리오 비교·모의 체결, 투자 일지)에서 사용자가 스스로 답·확인할 수 있는 질문만 만든다. 외부 자료(실적 발표 일정·수급 주체·업계 비교·뉴스)가 필요한 내용은 질문이 아니라 unknowns에 둔다 — 서비스가 답해줄 수 없는 확인을 유도해 신뢰 공백을 만들지 않는다. 강제 수단: 프롬프트 규칙 + 캐시 생성 시 수동 검수(기계 검증 불가 항목) |
| 금지 콘텐츠 | §7 사전 매칭 시 해당 블록 차단 + policy_result="blocked_partial" + 감사로그 |

**차단의 단위**: 위반 블록만 제거하고 나머지는 렌더(전체 실패 아님). 차단 건수는 안전 지표 카운터·`out/audit/`에 기록.

**동반자 대화(companion chat — D-0718-1120-main)**: `POST /api/companion/chat`의 응답도 본 §6 계약을 그대로 따른다(facts+source_id·as_of / interpretations 양면 / unknowns / next_questions, 렌더 전 guard 강제). 추가 필드 `reply_text`(말풍선용 평문 서술)를 허용하되 §7 사전 검사 대상이며, 숫자 포함 시 facts와 동일하게 허용 숫자 집합(스냅샷·엔진 산출)과 대조한다. 데이터 원천은 §3.1-b 실종목 fixture + §3.2 market_context 스냅샷의 **화이트리스트 필드만** 프롬프트에 격리 주입하고, 분할 비교·수수료 등 숫자는 엔진 계산 결과만 인용한다(LLM 산수 금지). 폴백 사슬: live → 사전 준비 문답 캐시(`data/fixtures/companion_cache/` — `[데모 고정]` 수기 검수 허용, 단 숫자는 엔진·스냅샷 산출값) → 안전 강등 문구("지금은 답변을 준비할 수 없어요 — 화면의 사실 카드를 참고해 주세요"). 실패를 숨기지 않는다.

## 9. 표기 계약 (화면·문구)

- **금액**: 원 단위 콤마(예: 459,011원). **비중·등락률**: 소수 1자리(%). **날짜**: `M/D(요일)` 또는 `YYYY-MM-DD`. **예외**: 출처 원천값을 직접 인용하는 사실 문장(동반자 대화의 시장 지표 등)은 원천 자릿수를 유지한다 — 재반올림은 산수 재가공으로 보아 금지(guard 숫자 대사와 정합, W2 판정 2026-07-18).
- **손익**: 색 + 부호(±) + 아이콘(▲▼) + 텍스트("수익/손실") **4중 병행**(색각이상 대응 — 장차법 §21·KWCAG 취지). 국내 관례색(상승 적/하락 청) 사용.
- **기준시각 문구**: 실동결 스냅샷 = `YYYY-MM-DD HH:mm KST 종가 기준` / fixture = `교육용 가상 데이터 (가상 기준시각 …)` — 두 배지를 시각적으로 구분.
- **모의 표시**: 주문·체결 화면(S0 주문 화면(재현)·⑥ 모의 주문·⑧ 주문 화면(재현)) 상단 고정 바 "모의 주문 — 실제 거래 아님".

- **재현 화면 표시 규칙(S0 진입·⑧ 종착 — 사용자 결정 2026-07-17, D-0717-2121)**: S0과 ⑧은 실앱 주문 화면의 재현(흐름 연결성 시연 전용)이다 — 주문 실행 UI는 ⑥ 모의 체결이 유일하며, 두 화면 모두 `data-role="order-execute"` 마커·"체결하기" 문구 금지, 상단 "실앱 주문 화면 재현(시연용)" 라벨 필수. **S0의 주문 버튼은 클릭 시 인터셉트 팝업(진입 갈림길)을 연다 — 주문 실행 아님.** ⑧의 주문 버튼은 `disabled`(브리핑 완료 후 종착 — 재인터셉트 없음). **구매/판매 버튼은 전 시나리오에서 병렬 노출하고 색 구분(구매 빨강/판매 파랑)만 한다 — side 강조·hidden 금지**(헌법 §14 "손실 화면 추가매수 기본 노출 금지"의 금지 대상은 유도 문구·기본 강조이지 중립적 버튼 존재가 아님 — 브리핑 후 저가 매수도 사용자의 정당한 선택지). ⑧에는 브리핑 재진입 배너를 두지 않는다(과정 재수행 유도는 번거로움 — 사용자 지적 2026-07-17) — 대신 **내 판단 기록 상기 카드**(내가 적은 투자 일지 인용·선택·다음 재검토 메모, 무기록 시 상태 문구)를 둔다. 앞 연결(레이어 진입) 재현은 S0 인터셉트 팝업이 담당한다.
