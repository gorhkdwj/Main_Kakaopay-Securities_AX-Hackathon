# 카카오페이증권 초보 투자자 매수·매도 지원 시스템 리서치 계획

- 작성일: 2026-07-16
- 상태: 검토 피드백 반영 완료 (2026-07-16) — Claude·Codex 이중 실행 및 교차검증 체계 반영
- 현재 범위: 계획 문서화만 완료하며, 리서치와 구현은 이후 진행합니다.

## 1. 목표와 범위

- 본 계획은 [본선 안내](../notice/2026-07-18_본선안내_카카오페이증권트랙.md)와 [인터뷰 분석](../research/2026-07-16_카카오페이증권_인터뷰영상분석.md)을 근거로, 7월 18일 당일 3시간 안에 문제 정의·기능 선택·동작 데모로 전환할 수 있는 의사결정 자료를 만드는 것이 목적입니다.
- 초기 문제 가설은 “초보 투자자가 고불안 매매 순간에 정보·위험·대안을 이해하지 못해 판단이 정체되는 문제를 해결하되, AI가 결론을 대신하지 않고 사용자가 매수·매도·보류를 스스로 선택하게 돕는다”로 설정합니다. 이는 결론이 아니라 리서치로 검증할 가설입니다.
- 국내·미국주식을 모두 조사하되, 본선 상세 요구사항과 데이터 안정성에 따라 한 시장의 핵심 시나리오만 데모합니다. 점수가 비슷하면 공식 데이터와 오프라인 폴백이 더 안정적인 시장을 선택하고, 동률이면 국내주식을 우선합니다.
- 실거래·수익 보장·확정적 매수·매도 추천·자율 주문은 사전 준비 범위에서 제외합니다. 주문이 필요하면 모의투자 또는 최종 확인 전 주문 미리보기까지만 다룹니다.
- 모든 워크스트림은 `조사 질문 → 방법 → 우선 출처 → 산출물 → 완료 기준` 순서로 관리합니다.

출처대장에는 다음 필드를 사용합니다.

```text
source_id, source_tier, source_type, organization_or_author, title,
published_at, accessed_at_kst, url, local_evidence_path, locator,
market, journey_stage, claim_ids, evidence_summary,
fact_or_interpretation, reliability, limitations, status, rejection_reason
```

주장대장에는 다음 필드를 사용합니다. 보고서의 핵심 주장은 반드시 주장대장에 등재하며, 교차검증(WS9)은 이 주장 단위로 수행합니다.

```text
claim_id, claim_statement, claim_type(F/I/H/D), journey_stage,
supporting_source_ids, counter_source_ids, confidence(H/M/L),
status(채택/기각/쟁점), notes
```

출처 및 근거 기록 규칙은 다음과 같습니다.

- 열어본 모든 자료에 ID를 부여합니다. **ID는 에이전트 접두어를 사용해 병합 시 충돌을 방지합니다**: Claude는 `C-SRC-0001`/`C-CLM-0001`, Codex는 `X-SRC-0001`/`X-CLM-0001`. 워크스트림별 번호 대역을 배정해 병렬 작업 시에도 충돌이 없게 합니다(WS1: 01xx, WS2: 02xx, WS3: 03xx, WS4: 04xx, WS5: 05xx, WS6: 06xx, WS7~9: 07xx~). 채택 자료뿐 아니라 중복·부적합·접근 실패 자료도 URL과 기각 사유를 남깁니다.
- 보고서의 사실·수치·인용·기능 설명에는 `[SRC-ID](정확한 딥링크)`를 인라인으로 붙입니다. AI 요약이나 검색 결과 페이지는 출처로 인정하지 않고 원문으로 역추적합니다.
- 근거를 `관찰 사실(F)`, `해석(I)`, `가설(H)`, `제품 결정(D)`으로 구분합니다. 핵심 문제 주장은 원칙적으로 공식·원논문 등 A급 원천 하나와 독립된 보조 근거 하나를 요구합니다.
- 앱 리뷰는 개별 딥링크가 없으면 앱스토어 목록 URL, 플랫폼·국가·앱 버전·별점·리뷰일·확인일·익명 ID를 기록하고 이를 한계로 표시합니다.
- 자체 인터뷰는 `url=N/A(자체 조사)`로 표시하고 익명 참여자 ID와 로컬 비식별 메모 경로를 기록합니다. 공개 검증 가능한 핵심 정량 근거로 사용하지 않습니다.
- PDF는 쪽·표 번호, 영상은 타임스탬프, 데이터는 조회 조건·기준시각·단위·분모·계산식을 기록합니다.
- 동적 페이지는 열람 시각과 화면 위치를 남기며, 계정·잔액·API 키 등 개인정보와 비밀정보는 캡처·로그·문서에 기록하지 않습니다.

## 4. 문제 재정의와 제품 선택 게이트

### 4.1 초보 투자자 정의

`첫 매수 준비형`, `보유 중 손실불안형`, `매도 판단 막막형`, `매도 후 재진입형`을 초기 가설로 두되, 최소 두 종류의 근거가 없는 세그먼트는 최종 타깃에서 제외합니다. 최종 정의에는 포함·제외 기준과 확신도를 명시합니다.

### 4.2 기회 우선순위

각 `세그먼트 × 의사결정 순간 × 페인포인트`를 다음 가중치로 평가합니다.

| 평가 항목 | 가중치 |
|---|---:|
| 근거 강도 | 25 |
| 심각도·발생 빈도 | 20 |
| 카카오페이증권 전략 정합성 | 20 |
| 규제·소비자보호 안전성 | 15 |
| 3시간 데모·데이터 실행성 | 15 |
| 기존 서비스 대비 차별성 | 5 |

규제상 적색이거나 출처·데이터를 안정적으로 제공할 수 없는 후보는 총점과 무관하게 탈락시킵니다. 최상위 후보를 핵심 데모, 다음 비중복 후보를 백업으로 선정합니다.

### 4.3 제품 경계

기본 형태는 출처 기반 대화형 의사결정 보조입니다. 사실·비용·위험·대안·시나리오를 설명하고 사용자에게 매수·매도·보류 선택권을 돌려줍니다. 개인화 추천, 종목 순위, 직접 주문은 규제 검토 결과에 따라 제한하거나 안전한 대체 UX로 전환합니다.

### 4.4 성공지표

지표는 **당일 시연 가능 지표**와 **설계 목표 지표**로 구분해 관리합니다. 브리프·시연에는 전자를 사용하고, 후자는 "실서비스라면 이렇게 측정하겠다"는 설계 의도로 제시합니다.

- **당일 시연 가능 지표** (3시간 데모에서 직접 보여줄 수 있는 것):
	- 안전 지표: 출처 없는 주장률(모든 응답에 출처 표시 여부), 중요 위험 누락률, 오래된 데이터 미표시율(기준시각 표기), 투자권유성 표현률, 주문 확인 실패율(확인 단계 동작 여부)
	- 과업 데모: 대표 시나리오의 과업 완료 흐름, 의사결정에 필요한 정보의 완결성
- **설계 목표 지표** (실서비스 가정의 측정 계획):
	- 사용자 지표: 정보 이해도, 위험 회상률, 판단 근거 설명 가능성, 과업 성공률, 의사결정 시간, 과신을 제외한 보정된 자신감
	- 비즈니스 지표: 셀프서비스 해결률, 상담·민원 감소 가능성, 신뢰·건강한 유지율
- 거래 횟수와 단기 수익률은 단독 성공지표로 사용하지 않습니다.

## 5. 검증·완료 기준과 가정

- 공식 홈페이지의 현재 메뉴·하위 페이지는 100% 인벤토리화하며, 최근 회사소식은 제목 전체를 훑고 관련 본문만 정독합니다.
- 보고서의 사실·숫자·인용은 100% 출처 ID와 링크를 가져야 하며, 링크 오류·출처 없는 핵심 주장은 0건이어야 합니다.
- 리뷰는 코어 목표(카카오페이증권 80 + 토스증권 40)를 우선하고 확장분은 시간 허용 시 수집합니다. 테마 포화 시 조기 중단할 수 있으며, 적격 리뷰가 부족하면 무관한 리뷰로 채우지 않고 실제 표본 수와 한계를 기록합니다.
- LLM으로 리뷰를 분류할 경우 20% 층화 표본을 사람이 재검수하며 일치율 85% 미만이면 코드북을 수정하고 재분류합니다.
- 커뮤니티·앱 리뷰의 테마 비율을 전체 고객 비율처럼 표현하지 않습니다. 정량 규모는 공식 조사·통계로만 뒷받침합니다.
- 규제 결론은 현행 법령·감독기관 원문으로 확인하며, 면책문구만으로 위험 기능을 허용하지 않습니다.
- KIS 등 주문 API는 모의환경과 이용조건을 먼저 검증합니다. 계좌·신청·이용목적·시세 재배포 제약이 있으므로 사용 가능성을 가정하지 않습니다.
- 토요일 시장 폐장, 네트워크 장애, API 키 미발급에 대비한 전일 데이터 스냅샷과 모의 체결 폴백을 필수로 평가합니다.
- 앱 접근은 가능하지만 실제 금전 거래는 수행하지 않습니다. 사용자 검증은 휴리스틱 검토를 본안으로 하며, 실사용자 테스트는 선택 항목입니다. 미수행 시 `실사용자 미검증` 한계를 보고서에 명시합니다.
- 교차검증(WS9) 완료 기준: 두 보고서의 주장대장을 전수 대조하고, 모든 상충 항목이 판정 규칙에 따라 `채택/기각/쟁점` 중 하나로 처리되어 불일치대장에 기록되어야 합니다. 합산본의 핵심 주장에는 양측 확신도가 병기되어야 합니다.
- 본선 당일 상세 공지가 본 계획과 충돌하면 당일 공지를 우선합니다.
- Git 초기화·커밋 등 저장소 관리 작업은 본 계획에 포함하지 않습니다.
