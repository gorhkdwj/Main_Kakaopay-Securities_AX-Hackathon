# 구현 계획 · 판단 여권 (Decision Passport)

> **지위**: 실행 단계의 작업 순서·모듈 설계 기준. 처음부터 전체를 달리지 않는다 — 가장 작은 성공 단위(오프라인 fixture 완주)부터. 모든 지표·수식·스키마의 단일 기준은 [기준 계약 문서](../requirements-contract.md)이고, 제품 경계는 헌법 §14(리서치 확정 제약)다. 병렬화 시 헌법 §13(worktree·소유권 매트릭스)을 따른다. 검증 기준은 [검증 계획](validation-plan.md).

## 0. 아키텍처 (리서치 §8.3 확정 — 계산·생성 분리 × 폴백 3계층)

```text
[데이터 계층]
  data/fixtures/scenario_*.json   ← 최종 폴백(계층3): 오프라인 완주 보장
  data/snapshots/ + manifest.json ← 동결 스냅샷(계층1): 수집 성공분만 as_of 표시
        │
[결정론 계층]  src/engine/        ← 숫자의 유일한 생산자 (계층2: 로컬 모의체결)
  calculator.py  수수료·거래세·수령액·실현손익·잔여비중·D+2  → calculation_id 발급
  settle.py      모의 체결 처리(주문 검증·체결 기록)
        │
[정책 계층]    src/policy/
  guard.py       금지 표현·무출처 사실·수량 불일치·미래 수익률 차단 (렌더링 전)
  lexicon.py     금지/허용 표현 사전 v1 (계약 §7)
        │
[생성 계층]    src/briefing/
  contract.py    AI 응답 계약(JSON) 스키마·검증
  compose.py     fixture/스냅샷 → facts/interpretations/unknowns 조립
  llm.py         (S5) LLM 호출 + 실패 시 응답 캐시 폴백  ※ LLM은 산수 금지
        │
[표시 계층]    src/webapp/
  app.py         로컬 서버(FastAPI 또는 stdlib http — S4에서 확정)
  static/        화면 6장 + 안전 지표 카운터 + 안전모드 토글
        │
[포장]         src/.codex-plugin/ + src/skills/   (S7 — 예선 형식 대응 껍데기)
[감사]         out/audit/                          (게이트 실행 로그 — Git 제외)
```

**불변 원칙**: ① 숫자는 engine만 만든다(LLM 산수 금지) ② 화면에 그려지는 모든 사실은 guard를 통과한 AI 응답 계약 JSON 또는 calculation 결과다 ③ LLM·네트워크가 죽어도 카드·계산기·완주는 성립한다.

## 2. 단계 상세

### S1 · 데이터 계약·fixture 확정
- **왜**: 모든 화면·계산·테스트가 같은 데이터 정의를 쓰게 하는 뿌리. "동결 성공분만 기준시각 표시"(F-CLM-0099) 원칙의 구현 전제.
- **작업**: ① 계약 §3 스키마대로 `scenario_loss8.json`(핵심)·`scenario_first_buy.json`(백업 K2)·`scenario_profit15.json` 작성 — 가온전자(가상) 데이터는 계약 §5 골든값과 일치시킴 ② manifest 스키마·검증 스크립트(`scripts/data/validate_fixture.py`) ③ 계약의 데모 고정값(수수료 KRX 단일·절사·경고 임계값 — Decisionlog D-0716-0923-main)이 fixture에 반영됐는지 확인.
- **완료 조건**: 검증 스크립트가 3종 fixture 전부 통과, `scenario_loss8`로 [목업](../mockup/2026-07-17_판단여권_목업.html) S0~S8의 모든 표시값이 채워짐.
- **실패 시 중단점**: 계약 확정 불가 항목 발견 → 계약 문서에 `확인 필요` 명시 + 데모 고정값을 Decisionlog에 기록 후 진행(막히지 않음).

### S2 · 결정론 계산 엔진 (`src/engine/`)
- **왜**: "숫자는 계산기가, 말은 LLM이" 원칙의 심장. 시연 신뢰성의 근거.
- **작업**: `calculator.py` — `sell_preview(수량, 기준가, 평단, 보유수량, 총평가자산, 체결일)` → 계약 §4 계산 결과 JSON(+calculation_id). `biz_days.py` — D+2 영업일(주말 제외 — 공휴일 미반영은 계약 §11 한계 명시). `settle.py` — 모의 체결(수량 검증→체결 기록 생성).
- **완료 조건**: 계약 §5 **골든값 3세트**(유지/10주/30주 — 수수료 69·207원, 수령 459,011·1,377,033원, 실현 -40,989·-122,967원, 비중 28.2→18.8→0%) 재현 + 경계값(0주 거부·1주·전량·보유 초과 오류) pytest 통과.
- **실패 시 중단점**: 수식 불일치 발견 → **계약 문서 갱신이 먼저**, 코드 먼저 수정 금지(헌법 §5).

### S3 · 정책 가드 (`src/policy/`)
- **왜**: 안전 게이트(권유·단정·무출처 0건)를 "화면으로 증명"하는 장치 — 심사 차별 포인트.
- **작업**: `lexicon.py` — 계약 §7 금지 표현 사전 v1(방향 결론/단정·보장/수익 예측 카테고리별 패턴)+허용 경계(감정 인정 등). `guard.py` — AI 응답 계약 JSON 입력 → ① facts의 source_id·as_of 필수 검증 ② 금지 표현 매칭 ③ user_inputs 수량과 계산 입력 일치 검증 ④ 위반 시 해당 블록 차단+`policy_result` 기록(전체 실패가 아니라 위반 블록만 제거, 차단 로그를 안전 지표 카운터에 반영).
- **완료 조건**: [검증 계획](validation-plan.md) §2 안전 테스트셋 1차(차단 케이스 전건 차단, 정상 케이스 오차단 0) 통과.
- **실패 시 중단점**: 정상 케이스 오차단 발생 → 사전(lexicon) 세분화. 차단 실패 → 패턴 보강. 테스트 완화 금지.

### S4 · 웹 UI 6장 (`src/webapp/` — fixture 구동, LLM 없이)
- **왜**: LLM 이전에 "완주 가능한 데모"를 확보 — 최소 성공 단위.
- **작업**: 화면 6장 = ① 상황·계획 회상 ② 사실 브리핑(compose가 fixture에서 조립한 정적 계약 JSON 렌더) ③ 체크리스트+이해 확인 ④ 대칭 시나리오+미리보기(engine 호출) ⑤ 검토 의향+이유 기록 ⑥ 별도 모의 주문 화면→체결→회고. 공통: 상단 as_of·출처 배지, **안전 지표 카운터**, 안전모드 토글(주문 유도 숨김 화면), 접근성(색+부호+아이콘+텍스트, 버튼 라벨).
- **UI 제약(헌법 §14)**: 대화 내 주문 실행 버튼 금지, 검토 의향 4버튼 동일 크기·기본 강조 없음, 손실 화면 추가매수 기본 노출 금지, '원인' 단어 미사용('관련 사실').
- **완료 조건**: `scenario_loss8`로 8단계 클릭 완주(네트워크·LLM 0회 호출), 목업과 화면 구성 일치.
- **실패 시 중단점**: 프레임워크 이슈 시 stdlib http+정적 HTML로 강등(기능 우선).

### S5 · LLM 브리핑 결합 (`src/briefing/llm.py`)
- **왜**: 쉬운 언어 설명·질문 정리의 가치 — 단 실패해도 데모가 성립해야 함.
- **작업**: fixture 사실 → LLM에 "쉬운 설명·다음 질문" 생성 요청(수량·금액 산출 금지 프롬프트) → 계약 JSON 파싱·guard 통과 → 카드 렌더. 타임아웃/실패 시 `data/fixtures/llm_cache/`의 사전 생성 응답 사용. 공시 텍스트는 데이터로 격리(프롬프트 인젝션 방어 — 검증 케이스 포함). API 키는 `.env`만.
- **완료 조건**: 안전 게이트 전건 통과 + 인젝션 케이스 차단 + LLM 차단 상태에서 캐시 폴백 완주.
- **실패 시 중단점**: 품질 미달 시 S5 전체를 캐시 모드로 고정하고 데모 성립(핵심 가치는 구조·계산·가드).

### S6 · 스냅샷 수집·동결 (`scripts/data/collect_snapshot.py`)
- **왜**: "실수집 성공분만 기준시각 표시" — 데이터 정직성의 실행.
- **작업**: 7/17 18시 이후 pykrx로 가상 시나리오 매핑용 지표(지수·시장 통계 수준)와 검증용 종가 수집 → 성공 육안 확인 → manifest(수집 시각 KST·출처·SHA-256) 기록·동결. KRX CSV 수동 다운로드는 사용자 수행(검증본). **주의**: 데모 종목은 가상이므로 스냅샷은 "시장 상황 카드·검증 증빙"용이고 종목 가격은 fixture가 기준 — 혼용하지 않음(계약 §3).
- **완료 조건**: manifest 검증 스크립트 통과 또는 "수집 실패 — fixture 단독" 결정 기록.
- **실패 시 중단점**: 수집 실패 시 지어내지 않고 fixture 단독 + 화면 문구는 "교육용 가상 데이터"만.

### S7 · 플러그인 포장·최종 리허설
- **작업**: `src/.codex-plugin/plugin.json` + `src/skills/decision-passport/SKILL.md`(데모 실행·시나리오 로드 명령) — 본체(webapp·engine)를 호출하는 얇은 껍데기(예선 요강의 "핵심 로직 분리" 시사점). 리허설: Wi-Fi 차단 완주 1회, 안전 게이트 실행, 로그 수집 절차(모든 worktree `logs/` 복사) 점검, 브리프 초안과 시연 흐름 대조.
- **완료 조건**: 리허설 체크리스트 전항 통과([검증 계획](validation-plan.md) §6).

## 3. 병렬화 계획 (헌법 §13)
- S1 완료 후 **S2(wt/engine)·S3(wt/policy)·S4(wt/ui)는 상호 독립** — worktree 3개 분담 가능. 소유권: engine=`src/engine/`+`tests/engine/`, policy=`src/policy/`+`tests/policy/`, ui=`src/webapp/`. 공유 파일(계약·로그 3종)은 main만 수정.
- 인터페이스 고정(병렬 전 main이 확정): engine 함수 시그니처·계산 JSON 스키마, guard 입출력(계약 §4·§6이 그 자체로 인터페이스 명세).
- S5는 S3·S4 병합 후 main 단독. 브랜치는 즉시 commit·push, main 병합은 게이트(§10) 통과 후.

## 4. 기술 스택 (최소주의)
Python 3.12(이미 설치·pykrx 호환) / 웹: FastAPI+단일 HTML(불가 시 stdlib) / 테스트: pytest / LLM: 환경변수 키(제공자는 S5에서 확정 — 데모는 캐시 폴백이 항상 성립하므로 종속 없음) / 외부 CDN 0(오프라인 원칙, 목업과 동일).
