## 1. 구현 단위: Codex 플러그인 (본선에서도 유사할 것으로 추정)

예선 과제는 "기업의 공개·검증 가능한 문제를 해결하는 **Codex 플러그인**" 제출이었습니다.

- 플러그인 필수 요소: `src/.codex-plugin/plugin.json`
- plugin.json 외에 **실제로 동작하는 구성 요소 1개 이상** 필수:
  - 스킬(`skills/<이름>/SKILL.md`) — 가장 일반적
  - MCP 서버(`.mcp.json`)
  - 직접 작성한 실행 코드
- src 내부 구조는 자유이나, **Codex 플러그인으로 실행이 되어야 함**.

예선 제출 구조:

```
submission.zip
├── src/
│   ├── .codex-plugin/plugin.json   # 필수
│   ├── skills/<이름>/SKILL.md
│   ├── .mcp.json
│   └── 그 밖의 실행 코드와 설정 파일
├── README.md
└── logs/                            # AI 대화 로그 전체
```

공식 문서:

| 문서 | URL |
|---|---|
| 플러그인 개요 | https://developers.openai.com/codex/plugins |
| 플러그인 만들기 | https://developers.openai.com/codex/plugins/build |
| 스킬 작성법 | https://developers.openai.com/codex/skills |

## 2. 로그 규정 (본선 안내와 동일 — 계속 유지 확실)

- AI와 주고받은 대화 로그 **전체를 편집·발췌 없이** 제출 (md, txt, json, jsonl).
- **로그·결과물·질문지(브리프) 사이의 정합성을 평가** — 어긋나면 불리.
- ⚠️ **로그를 사후 가공(편집·발췌·삭제)하여 제출하면 실격.**
- 수집 도구: log-hooks.zip (✅ 본 프로젝트 설치·동작 중) / 기타 도구는 텍스트로 무편집 저장.
- AI 채팅에 API 키·비밀번호·토큰 입력 금지.

## 3. 서술 요구: 질문 5문항 → 본선 브리프와 대응

예선 질문 5문항이 본선 브리프 필수 항목과 거의 1:1로 이어집니다. 심사가 무엇을 묻는지에 대한 힌트입니다.

| 예선 질문 5문항 | 본선 브리프 필수 항목 |
|---|---|
| 무엇을, 누가, 어떤 상황에서 쓰나요? | ① 한 줄 요약 |
| 왜 이 문제를 선택했나요? | ② 문제 정의 |
| 플러그인은 어떻게 작동하나요? | ③ 핵심 기능·시연 포인트 |
| AI를 어떻게 활용했나요? | ④ 사용 도구·데이터 출처 |
| 어떻게 검증했나요? | (선택) 한계·미구현 + 로그 정합성 |

## 4. 평가·근거 규정

- 예선은 **AI와 심사자가 함께 평가** (본선도 AI 사전 검토 가능성을 염두).
- **출처는 공개 자료로, AI가 확인할 수 있어야 함.**
  - 기업 내부 비공개 정보, 확인 불가한 사적 경험, 출처 없는 숫자는 근거 불인정.
- ❗️ **예선 제출물은 본선 심사에 활용될 수 있음** — 그룹바이에 업로드된 예선 결과물과 본선 산출물의 연속성·일관성을 신경 쓸 것.

## 5. 본선 설계 시사점

1. **실행 가능성이 전제 조건** — 예선에서 "실행되는 플러그인"이 최소 요건이었듯, 본선도 시연(1대1 심사, 본인 노트북)이 핵심이므로 **동작하는 데모**가 최우선. 문서·기획만으로는 부족.
2. **구현 스택 후보** — 본선도 Codex 플러그인(스킬 + MCP + 실행 코드) 단위가 유력. 스킬/MCP 구성 패턴을 미리 손에 익혀둘 것. 단, 형식이 바뀔 수 있으니 핵심 로직은 플러그인 껍데기와 분리해 두는 것이 안전.
3. **데이터 출처는 공개·검증 가능한 것만** — 주가·종목 데이터 등은 AI가 확인 가능한 공개 출처(공식 API, 공시 등)를 쓰고 출처를 브리프에 명시.
4. **검증 스토리를 준비** — "어떻게 검증했나요?"가 상설 질문. 테스트·검증 과정을 로그에 남기고 시연 포인트에 포함.
5. **로그가 곧 평가 자료** — 문제 정의→분해→해결→검증 흐름이 로그에 드러나도록 작업 순서를 설계 (인터뷰 영상 분석의 "과정의 설계" 평가 기준과 일치).
