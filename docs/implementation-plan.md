## 단계 상세 (핵심 2단계 예시 — 나머지는 착수 시 같은 형식으로 채움)

### S1 · 데이터 계약·fixture 확정
- 왜 필요한가: 모든 화면·계산·테스트가 같은 데이터 정의를 쓰게 하는 뿌리. 리서치 확정 원칙("동결 성공분만 기준시각 표시")의 구현 전제.
- 이전 단계와 연결: requirements-contract의 `확인 필요` 3건(수수료 시장 고정 여부·반올림·경고 임계값)을 여기서 확정하고 문서를 먼저 갱신.
- 다음 단계로 전달: fixture 3종·manifest 스키마 → S2 테스트 데이터, S4 화면 데이터.
- 완료 조건: scenario_loss8.json으로 목업 S0~S8의 모든 표시값이 채워짐.
- 검증 방법: 스키마 검증 스크립트(필수 필드·타입) 통과.
- 실패 시 중단점: 계약 확정 불가 항목은 `확인 필요`로 남기고 데모 고정값을 Decisionlog에 기록.

### S2 · 결정론 계산 엔진
- 왜 필요한가: "숫자는 계산기가, 말은 LLM이" 원칙의 심장. 시연 신뢰성의 근거.
- 이전 단계와 연결: S1 계약의 수식·반올림 규칙 구현.
- 다음 단계로 전달: calculation_id 붙은 결과 JSON → S4 표·S5 응답 계약의 calculation_id.
- 완료 조건: 목업 검증값(10주: 수수료 69·거래세 920·수령 459,011·비중 28.2→18.8% / 30주: 수령 1,377,033) 재현 + 경계값(0주·전량·1주) 테스트 통과.
- 검증 방법: pytest — 정상·경계·오류(수량 초과) 케이스.
- 실패 시 중단점: 수식 불일치 발견 시 계약 문서 갱신이 먼저(코드 먼저 수정 금지).

## 병렬화 가능 지점 (worktree 분담 후보)
- S2(engine)·S3(policy)·S4(ui)는 S1 완료 후 상호 독립 — 세션 분담 가능(소유권: src/engine, src/policy, src/webapp).
- S5는 S3·S4 병합 후 단일 세션(main) 수행 권장.
