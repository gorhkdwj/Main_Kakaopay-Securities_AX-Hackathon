# AGENTS.md · 판단 여권(Decision Passport) 작업 헌법

이 문서는 이 프로젝트에서 AI 협업 에이전트가 **매번 지시하지 않아도 항상 따라야 하는 기준**이다. 사용자와 에이전트 모두 이 규칙을 기준으로 일한다. 규칙이 바뀌면 이 문서를 먼저 갱신하고 Decisionlog에 이유를 남긴다.

## 헌법 동기화 규칙
- `CLAUDE.md`와 `AGENTS.md`는 동일한 공통 작업 헌법을 Claude와 Codex에 제공하는 도구별 진입점이다.
- 공통 규칙을 변경할 때는 두 파일을 같은 작업 단위에서 함께 수정하고, `README.md`의 헌법 안내와 Worklog를 함께 갱신한다. 방향을 바꾸는 변경이면 Decisionlog에도 이유를 남긴다.
- 도구별로 다르게 적용해야 하는 규칙만 해당 파일의 `[Claude 전용]` 또는 `[Codex 전용]` 블록에 둔다. 그 외 제목을 제외한 공통 본문은 동일하게 유지한다.
- 두 파일의 공통 규칙이 어긋나면 사용자의 최신 명시 지시와 Decisionlog를 기준으로 즉시 정합화한다. 도구 전용 블록은 명시된 도구에만 적용한다.

### [Codex 전용]
- 프롬프트 로그 제출이 필요한 Agent Coding은 검증된 Codex CLI를 기본 실행 표면으로 사용한다. VS Code·Antigravity 확장 프로그램은 `logs/codex`에 완료 턴이 원본과 일치함을 세션별로 검증한 경우에만 공식 작업에 사용한다.
- 새 로컬 clone에서는 프로젝트 신뢰 후 `/hooks`에서 저장소의 `.codex/hooks.json`을 1회 검토·승인한다. 훅 승인 전 작업은 자동 기록된 것으로 간주하지 않는다.
- 완료 보고 전 현재 세션의 `logs/codex/<session_id>.jsonl`이 유효한 JSONL이며 완료된 사용자·에이전트 대화가 원본과 순서·내용까지 일치하는지 확인한다. 진행 중인 현재 턴은 `Stop` 훅 실행 전이므로 기록 완료로 보고하지 않는다.

## 0. 프로젝트 정보
- 프로젝트 이름: 판단 여권 (Decision Passport)
- 목적: 초보 투자자가 급락·손실 등 고불안 순간에 출처 있는 판단 재료를 보고 스스로 설명 가능한 매수·매도·보류 결정을 내리도록 돕는 데모 시스템 (AX 인재전쟁 본선, 카카오페이증권 트랙)
- 주요 사용자: 초보 투자자(연령이 아닌 경험·역량 축 정의 — 리서치 §1.3). 직접 청중은 본선 심사위원(1대1 노트북 시연)
- 최종 산출물: 로컬 웹앱 데모(본체) + Codex 플러그인 포장(병행) + 오프라인 fixture 폴백 + 결과물 브리프 + 프롬프트 로그 전량
- 마감·제약: 2026-07-18(토) 본선 3시간(10:00~13:00), 12:00 초반 1차 제출(선착순 심사), 시장 폐장(실시간 시세 없음), 실주문·개인화 추천 금지, **본선 당일 공지가 본 문서·계획과 충돌하면 당일 공지 우선**
- 현재 사용자 수준: AI 도구 활용에 능숙한 실무형 참가자
- 원하는 진행 방식: 검증 중시(리서치의 근거·확신도 체계 계승), 격식 한국어(합쇼체), 결론 먼저

## 1. 대화와 보고 방식
- 결론을 먼저, 근거·세부는 뒤에.
- 기술 용어는 처음 등장할 때 한 줄로 설명한다.
- 불확실하면 단정하지 말고 `확인 필요`로 표기한다.
- 실행/확인하지 않은 것을 완료라고 말하지 않는다.
- 효과 주장(불안 감소·문의 감소 등)은 검증 전에는 항상 "가설"로 구분해 말한다(리서치 확신도 체계: H/M/L).

## 2. 작업 시작 전 확인
- 현재 폴더 구조와 기존 파일을 먼저 확인한다.
- 기존 파일·변경사항을 함부로 덮어쓰지 않는다.
- 바로 구현하지 말고, 영향 범위와 계획을 먼저 정리한다.
- 리서치가 확정한 제약(§14)과 Decisionlog의 기존 결정을 먼저 확인한다.
- **모든 구현·설계 작업은 계획 문서 기반으로 수행한다**: 무엇을·왜는 `docs/plans/project-plan.md`, 어떤 순서·어떤 모듈은 `docs/plans/implementation-plan.md`, 어떻게 검증은 `docs/plans/validation-plan.md`, 지표·수식·스키마·표기의 단일 기준은 `docs/requirements-contract.md`. 계획·계약과 다르게 작업해야 하면 **해당 문서를 먼저 갱신**하고(방향 변경이면 Decisionlog 기록) 그 다음 구현한다.

## 3. 파일 구조 원칙
- `src/` 실행 코드(웹앱 본체 + `src/.codex-plugin/` 플러그인 포장) · `tests/` 테스트·안전 테스트셋 · `tools/` 보조 스크립트(로그 훅 포함 — 수정 금지) · `scripts/` 데이터 수집·검증 스크립트 · `docs/` 문서(계획 3종은 `docs/plans/`) · `data/snapshots/` 동결 스냅샷 · `data/fixtures/` 오프라인 시나리오 · `out/` 실행·임시 산출물(Git 제외).
- `logs/`는 프롬프트 로그(해커톤 제출물)다 — **§13-4 불가침 규칙** 적용.
- 필요 없는 폴더는 만들지 않고 이유를 남긴다.

## 4. 구현 원칙
- 처음부터 전체를 달리지 않고 가장 작은 성공 단위부터 만든다(오프라인 fixture로 완주 → 스냅샷 연결 → LLM 설명 결합 순).
- **계산·생성 분리**: 숫자(체결·비용·세금·정산·잔여비중)는 결정론 계산 엔진이 만들고, LLM은 근거가 붙은 설명만 한다. LLM이 산수를 하게 두지 않는다.
- 각 단계는 다음을 명확히 한다: 왜 필요한가 / 이전 단계와 연결 / 다음 단계로 전달 / 완료 조건 / 검증 방법 / 실패 시 중단점.
- 단계 완료 보고에 포함: 전체 중 현재 위치, 이번에 한 일, 변경 파일, 검증 결과(**§6 단계 정합 QA 결과 — 직전·직후·전체, 발견·수정 내역 포함**), 남은 단계, 예상 잔여량, 발생한 문제·T-ID, 필요 시 Git commit/push 여부.

## 5. 기준 계약 문서 원칙
- `docs/requirements-contract.md`가 지표·수식·스키마·표기·판정의 **단일 기준**이다: 골든값(§5.2)·안전 지표 분모(§10)·금지 표현 사전(§7)을 포함하며, 코드 상수·화면 표기·테스트 기대값은 전부 여기서 나온다.
- 같은 지표가 파일마다 다르게 정의되지 않게 한다. 기준이 바뀌면 코드보다 계약 문서를 먼저 갱신하고 이유를 기록한다. 계획 3종(`docs/plans/`)의 수치·완료 조건이 계약과 어긋나면 **계약이 우선**하고 즉시 정합화한다.
- `[데모 고정]` 값(계약 §13 대장)은 실서비스 전 재검토 대상임을 항상 병기한다.

## 6. 테스트와 검증 원칙
- **단계 정합 QA 루프(모든 작업 단계 필수 — 결정: D-0716-0952-main)**: 어떤 작업 단계든 완료 보고 전에 3방향 정합을 스스로 검토·검증하고, 어긋남을 발견하면 **보고 전에 자체 수정**한다.
  - ① **직전 정합**: 이번 작업이 직전 단계 산출물의 가정·인터페이스를 깨뜨리지 않았는가(입력 스키마·기존 테스트·이미 병합된 코드와 충돌 없음).
  - ② **직후 정합**: 다음 단계가 기대하는 전달물(완료 조건·인터페이스·데이터)이 계획대로 충족되는가(implementation-plan의 "다음 단계로 전달" 항목 대조).
  - ③ **전체 정합**: 계획 3종(`docs/plans/`)·기준 계약(`docs/requirements-contract.md`)·헌법 §14 제약·기존 Decisionlog 결정·전체 데모 흐름과 충돌이 없는가.
  - 자체 수정으로 해소되지 않는 어긋남(방향 충돌·계약 변경 필요)은 작업을 멈추고 계약/계획 갱신 또는 Decisionlog·T-로그로 올린 뒤 진행한다. QA 수행 사실과 발견·수정 내역은 완료 보고의 검증 항목에 명시한다(발견 0건이어도 "정합 QA 통과" 명시).
- 영향 범위에 맞춰 검증한다: 단위·통합·입력·오류 상황·경계값 검증, 문서-코드 정합성, README-기능 일치, 제출물 구조 확인.
- **안전 게이트**(출시·시연 전 필수): 출처 없는 사실 0건, 권유·단정 표현 0건, 기준시각 누락 0건 — 사전 정의 테스트셋과 분모를 함께 기록한다(`docs/validation-plan.md`).
- 검증하지 못한 부분은 숨기지 말고 `미검증 범위`로 보고한다. 실패한 테스트를 삭제·완화해 통과시키지 않고 원인을 수정한다.

## 7. 보안·개인정보·비밀정보 원칙
- 다음을 코드·문서·로그·테스트 데이터·**AI 채팅**에 넣지 않는다: API 키, 비밀번호, 액세스 토큰, 세션 쿠키, 실제 고객·계좌정보. (해커톤 규정: 채팅에 넣은 비밀정보는 제출 로그에 그대로 남는다)

- 원본 데이터(스냅샷·리뷰 원문 등)는 수정하지 않는다. 필요하면 파생 파일을 별도로 만든다.
- 비밀정보가 발견되면 임의로 로그를 편집하지 말고, 노출 범위와 안전한 교체·폐기 방법을 사용자에게 보고한다(로그 편집 = 실격 사유).
- 데모 데이터는 가상 종목만 사용한다(실존 종목 사용 시 권유 오해 소지).

## 10. Git 원칙 (결정: D-0716-0903-main)
- **브랜치에서는 자유롭게, main은 게이트로**: 각 세션은 자기 브랜치(`wt/<주제>`)에서 작은 단위로 **즉시 commit·push**한다(백업+타 세션 가시성, WIP 허용). main 병합만 검증 게이트를 통과해야 한다.
- **병합 게이트(작업 단위 완료 시)**: ① implementation-plan의 해당 단계 완료 조건 충족 ② 테스트 통과(엔진·가드 등 영향 범위) ③ 계약 문서-코드 정합 ④ Worklog 기록 존재 ⑤ push 전 비밀정보 스캔(.env 외부 노출·키 문자열) ⑥ main 최신분 반영 후 재검증. 병합은 조율 세션(main)이 수행한다.
- **PR vs 로컬 merge**: 사전 준비 기간은 GitHub PR(기록·리뷰 뷰), **본선 당일 3시간은 로컬 merge**(속도 우선 — PR 왕복 생략).
- push는 비밀정보의 **회수 불가 지점**이다 — 브랜치라도 push 전 스캔을 습관화한다.
- 로그(`logs/`), 비밀정보(`.env`), 임시 산출물(`out/`)은 Git에 올리지 않는다. 단 로그 훅 설정(`.claude/`, `.codex/`, `tools/save_log.py`)은 추적 상태를 유지한다 — worktree마다 훅이 살아 있어야 한다.
- 커밋 메시지는 한 줄 요약 + 필요 시 본문. 세션 슬러그(§13)를 본문에 남긴다.

## 11. 로그 기록 규칙 (운영 로그 3종)
ID는 **타임스탬프 기반**이다(§13-2의 멀티 세션 규약): `W-MMDD-HHMM[-세션슬러그]` 형식 (D-, T- 동일). 예: `D-0716-0853-main`. 같은 분에 두 건이면 초를 붙인다. 순번(번호) 재부여는 하지 않는다 — 시간순 정렬이 ID에 내장되어 있다.

### Worklog.md — 주요 요청마다
모든 주요 사용자 요청이 끝날 때마다 기록한다. 최신 항목을 위에 추가한다(append-only).
```
### W-MMDD-HHMM-세션 · 작업 제목
**요청** / **수행 작업** / **변경 파일** / **검증**(미실행 시 사유) / **판단 근거** / **결과**(완료·남은 작업)
```

### Decisionlog.md — 방향을 바꾸는 결정만
대상: 문제 선정, 기술 스택, 구현 범위 확정·축소, 데이터 처리, 보안·개인정보, 테스트·검증 방식, 제출·공유 방식, 큰 구조 변경. 사소한 수정은 Worklog만.
```
### D-MMDD-HHMM-세션 · 결정 제목
**상황** / **검토한 선택지** / **결정** / **근거** / **영향** / **재검토 조건**
```

### Troubleshootinglog.md — 실제 문제 발생 시
실제 오류·실패·환경 문제·검증 실패·설계 충돌이 발생하면 기록한다. 같은 문제가 반복되면 새 T-ID 전에 기존 T-ID를 먼저 확인한다.
```
### T-MMDD-HHMM-세션 · 문제 제목
**발생 상황** / **증상** / **확인된 원인**(+불확실점) / **조치**(+최종 해결) / **재발 방지**
```

## 13. 멀티 세션·worktree 협업 규약
여러 에이전트 세션이 동시에 작업할 때의 규칙이다. (결정: Decisionlog `D-0716-0853-main`)

1. **작업 격리**: 병렬 세션은 각자 git worktree + 브랜치(`wt/<주제>`)에서 작업한다. main 직접 작업은 단일 세션일 때만.
2. **운영 로그 ID = 타임스탬프 + 세션 슬러그**: §11 형식. 순번이 아니므로 병합 시 재번호가 없고, 서로 다른 세션의 기록이 ID 충돌 없이 시간순으로 꿰어진다. 세션 슬러그 = 브랜치 끝 이름(main, engine, ui 등).
3. **운영 로그 병합**: 로그 3종은 append-only로만 쓴다. 병합 충돌 시 **양쪽 블록을 모두 보존하고 타임스탬프순으로 정렬**한다(내용 편집 금지). 필요하면 병합 후 파일 상단 목차(INDEX)에서 표시용 일련번호를 부여한다 — 표시 번호는 뷰일 뿐, 참조는 항상 타임스탬프 ID로 한다.
4. **프롬프트 로그(`logs/`) 불가침**: 병합·정렬·넘버링·편집 절대 금지(무편집 제출 규정 — 위반 시 실격). `logs/`는 gitignore되어 있어 **각 worktree에 각자 쌓인다** — 제출 전 모든 worktree의 `logs/`를 한 폴더로 **복사만** 하여 수집한다(파일명이 세션 UUID라 충돌 없음). worktree를 지우기 전에 반드시 로그부터 수집한다.
5. **파일 소유권 매트릭스**: 병렬 작업 시작 시 조율 세션(main)이 세션별 담당 경로를 Worklog에 선언한다(예: engine 세션=`src/engine/`+`tests/engine/`, ui 세션=`src/webapp/`). 공유 파일(로그 3종, README, 계약 문서)은 append-only 또는 main 세션만 수정. 담당 밖 파일이 필요하면 먼저 Worklog에 요청을 남긴다.
6. **병합 순서**: 브랜치 병합은 main(조율 세션)이 §10의 병합 게이트를 통과시킨 뒤 수행한다 — 브랜치 내 commit·push는 게이트 없이 자유. 코드 충돌은 소유권 매트릭스로 예방하는 것이 원칙이고, 발생하면 T-로그에 기록 후 해결한다.

## 14. 리서치 확정 제약 (구현 시 항상 적용)
리서치 최종 보고(`docs/report/2026-07-17_리서치_최종보고.html`)가 확정한 제품 경계다. 변경하려면 Decisionlog에 결정을 남긴다.
- **하지 않는다**: 개인화 종목 추천, 방향 결론("지금 파세요"), AI 자동·실주문, 대화 내 주문 실행 버튼, 수익 예측·보장, 게임화(콘페티·뽑기), 실시간 API 의존, 손실 화면의 추가매수 기본 노출.
- **항상 한다**: 모든 사실에 출처·기준시각 배지, 사실/해석/모름 분리, 대칭 선택지(기본 강조 없음), 비용·세금·D+2·비가역성 사전 고지, 별도 화면 모의 체결, 단정 표현 가드, 장애 안전모드, 접근성(색+부호+아이콘 병행).
- **AI 응답 계약**: facts(+source_id·as_of) / interpretations / unknowns / user_inputs / calculation_id / policy_result / next_questions — source_id 없는 사실·미래 수익률·매매 결론·사용자 입력과 다른 수량은 렌더링 전 차단.
- **데이터**: 실제 수집 성공을 확인하고 동결한 스냅샷만 "기준시각"으로 표시한다. 미래 시각을 예정 사실처럼 쓰지 않는다.
